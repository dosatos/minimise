## 💡 Learnings & Gotchas

### What Worked Well ✅
1. **Subagent-driven development** — 7 tasks executed smoothly with quality gates (spec + code review)
2. **TDD approach** — Every task started with tests; caught bugs early
3. **Quality gates** — 2-stage review (spec compliance + code quality) prevented issues from propagating
4. **Bug detection & fixes** — Task 4 database state invariant issue found in review, fixed in separate cycle
5. **Modular architecture** — Each component (DB, Git, Executor, API) is independent and testable
6. **Clear interfaces** — Each task knew exactly what inputs/outputs to expect from prior tasks

### Key Decisions
- **SQLite over JSON files** — For ACID guarantees and concurrent access safety
- **Base commit tracking** — Diffs calculated against base, not between adjacent tasks (simpler, more reliable)
- **Default handover (prompt)** — Not configurable per-task; simplified design
- **Non-blocking CLI** — REST API runs in separate thread, CLI doesn't wait for jobs to complete
- **No continue-on-error** — Jobs fail on first task failure (keeps orchestration simple)

### Gotchas to Watch
1. **Git state validation is strict** — Repository MUST be clean before creating jobs. Users need to commit/stash changes first.
2. **Database locking** — SQLite can have contention under heavy concurrent load. Monitor if parallel execution is added.
3. **Line counting in diffs** — Current handover manager counts lines starting with `+/-`, which catches false positives in code. Acceptable for heuristic purposes.
4. **Task output size** — Very large task outputs could impact token efficiency in handover. Consider truncation if needed.
5. **Claude Code invocation** — Currently via `npx claude-code -p context_file`. Depends on Claude Code being installed.

### Patterns to Repeat
- Task-specific review checklist (spec + quality) works well
- Fresh subagent per task prevents context pollution
- Commit early and often (small, focused commits)
- Test-first development (TDD) catches bugs before review
- Exact code in plan specs (not pseudocode)
