1. Problem Statement

Minimise currently hardcodes ClaudeCodeHarness as the execution backend for tasks and loops (e.g., in TaskExecutor and LoopEngine). While the codebase correctly defines an abstract AgentHarness interface, the lack of configuration and a concrete implementation for other harnesses prevents users from orchestrating tasks with alternative capable agents.

Specifically, we want to add support for the Pi coding agent (pi), an open-source agent harness with multi-provider support (Google, Anthropic, OpenAI). This allows users to leverage pi's model flexibility, JSON streaming output, and tool control within Minimise's deterministic orchestration, refinement loops, and hook ecosystems.

Pi's --mode json event stream is documented at agent-types.ts. The key event types we consume are: message_start, message_update, message_end, and agent_end. This is structurally different from Claude's stream-json — see §6.2 for the correct parsing strategy.

2. High Level Approach

To introduce pi support while keeping the system extensible, we will:

  1. Add wrap_prompt() to AgentHarness: A new optional method on the abstract interface that lets each harness inject its own prompt preamble (e.g. Claude's "do not create exploratory jobs" warnings, pi's tool-use instructions). This is the seam that moves harness-specific prompt text out of TaskExecutor._invoke_agent.
  2. Decouple Harness Selection: Introduce a HarnessFactory that resolves the active harness from the resolution hierarchy (task/step → persona → CLI → global settings). The factory accepts context (step_type, worker, persona) and returns an AgentHarness instance. This replaces the hardcoded ClaudeCodeHarness() calls in TaskExecutor and LoopEngine.
  3. Implement PiHarness: Create a new class PiHarness(AgentHarness) in src/minimise/agents/harness.py.
    • Manage the pi subprocess lifecycle via subprocess.Popen (same threaded stdin/stdout/stderr pattern as ClaudeCodeHarness).
    • Implement _build_env() to pass through multi-provider API keys while isolating the environment.
    • Map Minimise's abstract arguments (model, system_prompt, allow_edits) to pi's CLI parameters.
  4. Log Streaming & Parsing (corrected): pi's --mode json emits a different JSON structure than Claude's stream-json. pi uses "type":"message_start" / "type":"message_update" / "type":"message_end" events with nested assistantMessageEvent deltas (including thinking_delta and text_delta). We need a separate _extract_text_pi() function that:
    • Filters to message_end events (for live-logging, parse text_delta events in message_update)
    • Extracts content[].text blocks where role == "assistant"
    • Filters out content[].type == "thinking" blocks (reasoning models)
    This ensures mini job logs --query filtering and LoopEngine JSON control line stripping function correctly with both harnesses.
  5. Lifecycle & Telemetry: PiHarness yields HarnessResult with accurate exit_reason values (success, agent_error, timeout), matching the exact contract required by Minimise.

3. AgentHarness Interface Extension

The existing AgentHarness interface needs one new method to support harness-specific prompt extensions without leaking harness knowledge into TaskExecutor:

class AgentHarness(ABC):
    @abstractmethod
    def run(self, prompt, *, cwd=None, timeout=None, model=None,
            system_prompt=None, allow_edits=False, log_path=None,
            log_fields=None, log_filter=None) -> HarnessResult:
        raise NotImplementedError

    def wrap_prompt(self, prompt: str) -> str:
        """Optional: inject harness-specific instructions into the prompt.

        The default is a no-op. ClaudeCodeHarness overrides this to prepend
        the "do not create exploratory jobs" / "no co-author trailers" warnings;
        PiHarness leaves it as a no-op (those warnings are Claude-specific).
        """
        return prompt

This method is called by TaskExecutor._invoke_agent (renamed from _invoke_claude_code) before passing the prompt to self.harness.run(). The Claude-specific warnings move from TaskExecutor into ClaudeCodeHarness.wrap_prompt(). PiHarness inherits the no-op default — pi doesn't need those warnings.

Backward Compatibility: The default no-op means all existing code continues to work. Only TaskExecutor changes (one new call before harness.run()). The abstract run() interface is unchanged.

4. Model String Translation

A shared persona registry means the same model field might be consumed by different harnesses with different model-string formats:

Harness Format Example
claudeBare model IDclaude-sonnet-4-8
piProvider-prefixedanthropic/claude-sonnet-4-8

Strategy: Personas and settings use pi-style model strings (provider-prefixed) as the canonical format. Each harness's run() method translates the canonical string to its own format:

# ClaudeCodeHarness.run(): strip the provider prefix
if model and "/" in model:
    model = model.split("/", 1)[1]   # "anthropic/claude-sonnet-4-8" → "claude-sonnet-4-8"

# PiHarness.run(): use as-is (pi accepts provider/id format via --model)

This translation is internal to each harness class — no external mapping table, no factory responsibility. The persona registry stores one canonical string. If a persona is used with ClaudeCodeHarness, the harness strips the prefix. If used with PiHarness, it passes through unchanged. This is the simplest approach (no global registry, no cross-cutting concern) and makes future harnesses trivial: each owns its own translation.

A bare model string (no /) is passed through unchanged by both harnesses — treated as "the user knows what they're doing."

5. Configuration & Hybrid Harness Execution

5.1 Configuration File: ~/.minimise/settings.yaml

A single new config file (alongside the existing personas.yaml). Schema:

# ~/.minimise/settings.yaml  (all keys optional)
version: "0.0.1"
harness: claude         # default harness for all tasks/steps (optional, default: claude)
model: null             # default model, provider-prefixed (optional, default: none)

Why settings.yaml and not config.yaml: The codebase already uses ~/.minimise/personas.yaml. Settings (harness defaults) are distinct from personas (agent personality prompts). Calling it config.yaml would be ambiguous alongside the existing files. settings.yaml follows the pi convention and is unambiguous.

5.2 Hierarchical Resolution

When Minimise resolves the active harness for a task/step, it applies the following precedence (highest to lowest):

  1. Task/Step Explicit Harness: Specified in the job plan or loop spec (e.g., harness: pi on a PlanTask or Worker).
  2. Persona Default Harness: Defined in ~/.minimise/personas.yaml. If a task assigns a persona, the execution inherits the persona's harness field.
  3. CLI / Env / Global Default: --harness CLI flag → MINIMISE_HARNESS env var → settings.yaml harness: — whichever is set first, resolved once at job start / loop start time.
  4. Hardcoded Default: claude (preserves today's behavior for users who configure nothing).

5.3 Model Schema Changes for Per-Task/Per-Step Harness

The following model fields are added (all optional, all default None = inherit from higher-level config):

Model New Field Persistence
PlanTask harness: Optional[str] Plan YAML (no DB change — plans are files)
Worker (loop plan/implement/dimension) harness: Optional[str] Loop spec YAML (no DB change)
Persona harness: Optional[str] personas.yaml
Task (dataclass) harness: Optional[str] New harness TEXT column in tasks table (DB migration)

DB migration: A single ALTER TABLE tasks ADD COLUMN harness TEXT in Database.init_db(). The column is nullable — existing rows stay NULL (inherit from config). The Task.to_dict() serializer adds "harness": self.harness for API visibility.

Why Task needs it in the DB: A resumed job must preserve the harness choice that was in effect when mini job start was first called. Without storing it with the task, a resume after crash would re-resolve from config and could pick a different harness. The DB column mirrors how assignee and base_commit are already persisted — same pattern.

5.4 HarnessFactory Interface

class HarnessFactory:
    """Instantiate an AgentHarness for a Task or Worker."""

    def __init__(self, personas=None, *, default_harness: str = "claude",
                 default_model: Optional[str] = None,
                 backend: Optional[JobLogBackend] = None):
        ...  # delegates to internal _NameResolver

    def for_task(self, task) -> AgentHarness:
        """Resolve + instantiate a harness for a Task (job plan).
        Order: task.harness/model → persona.harness/model → factory default."""

    def for_worker(self, worker) -> AgentHarness:
        """Resolve + instantiate a harness for a LoopEngine Worker.
        Order: worker.harness/model → persona.harness/model → factory default."""

    def resolve_prompt_for_task(self, task) -> Optional[str]:
        """Resolve a Task's system prompt via assignee persona."""

    def resolve_prompt_for_worker(self, worker) -> Optional[str]:
        """Resolve a Worker's system prompt: persona.prompt | prompt | prompt_file."""

LoopEngine per-step resolution: The engine holds a self.factory: HarnessFactory and calls self.factory.for_worker(worker) at the start of each step. This resolves harness and model from the worker's harness/model fields, falling back through persona defaults to the factory's own defaults.

TaskExecutor per-task resolution: Similarly, TaskExecutor holds a self._factory: HarnessFactory and resolves per-task in execute_task() via self._factory.for_task(task), which respects task.harness/task.model and task.assignee.

6. User Experience

  • Global Configuration: Set default harness in ~/.minimise/settings.yaml or via MINIMISE_HARNESS=pi.
  • Per-Job Override: 
    mini job start <ID> --harness pi
  • Provider & Model Flexibility: Model strings in personas/settings use pi's provider/model format (canonical). Each harness translates internally: 
    mini job start <ID> --harness pi --model openai/gpt-4o
  • Transparent Execution: Logs, Gantt charts, git diffs, and hooks function identically. The only difference is which binary executes the tasks.
  • Health Checks: A new mini doctor command verifies that the selected harness binary is installed and the chosen provider is authenticated.

7. PiHarness Implementation Details

7.1 CLI Invocation

cmd = ["pi", "--mode", "json", "-p", "--no-session",
     "--no-extensions", "--no-skills", "--no-context-files"]

# Model: pi accepts provider/id format natively via --model
if model is not None:
    cmd += ["--model", model]

# System prompt: replaces pi's default coding prompt entirely
if system_prompt is not None:
    cmd += ["--system-prompt", system_prompt]

# Tool control: explicit allowlist (no permissions gate in pi)
if allow_edits:
    cmd += ["--tools", "read,bash,edit,write,grep,find,ls"]
else:
    cmd += ["--tools", "read,grep,find,ls"]

Why no --thinking off: The parsers _extract_text_pi_live and _extract_text_pi_final already filter thinking blocks from logs and final output, so reasoning content is handled correctly regardless. Forcing --thinking off would break reasoning-native models (o1/o3) and degrade others unnecessarily. Let pi use its default thinking level — the model/provider decides.

7.2 JSON Output Parsing (corrected)

pi's --mode json event format is structurally different from Claude's stream-json. A single turn looks like:

{"type":"session","version":3,"id":"...","timestamp":"...","cwd":"..."}
{"type":"agent_start"}
{"type":"turn_start"}
{"type":"message_start","message":{"role":"assistant","content":[],...}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"thinking_start",...}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"thinking_delta","delta":"..."}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"thinking_end","content":"..."}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"text_start","contentIndex":1,...}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"text_delta","delta":"Hello"}}
{"type":"message_update","message":{...},"assistantMessageEvent":{"type":"text_end","content":"Hello world"}}
{"type":"message_end","message":{"role":"assistant","content":[...]}}
{"type":"turn_end","message":{...},"toolResults":[]}
{"type":"agent_end","messages":[...]}

Key differences from Claude's stream-json:

  • No "type":"assistant" event — it's "type":"message_start" with message.role == "assistant"
  • Text arrives as "type":"message_update" with nested assistantMessageEvent.type == "text_delta" (not flat)
  • assistantMessageEvent.type can be thinking_start/thinking_delta/thinking_end (must be filtered)
  • The final "type":"message_end" contains the complete accumulated message — the simplest extraction target
  • "type":"agent_end" carries the full conversation history

Parsing strategy: two extraction functions.

For live logging (per-chunk JSONL, enables mini job logs -f):

def _extract_text_pi_live(event: dict) -> str:
    """Extract assistant text from a single pi JSON event, for live streaming.

    Handles message_update events with nested assistantMessageEvent deltas.
    Filters out thinking blocks; only accumulates text_delta chunks.
    """
    if event.get("type") != "message_update":
        return ""
    ame = event.get("assistantMessageEvent")
    if not isinstance(ame, dict):
        return ""
    if ame.get("type") == "text_delta":
        return ame.get("delta", "")
    return ""  # thinking_delta, thinking_start/end → skip

For final output (the accumulated text that feeds HarnessResult.output):

def _extract_text_pi_final(event: dict) -> str:
    """Extract the FULL assistant text from a message_end event.

    This is called once at the end and replaces the per-chunk accumulation
    for HarnessResult.output. It reads the complete, accumulated message.
    """
    if event.get("type") != "message_end":
        return ""
    msg = event.get("message") or {}
    if msg.get("role") != "assistant":
        return ""
    content = msg.get("content") or []
    if not isinstance(content, list):
        return ""
    return "".join(
        block.get("text", "")
        for block in content
        if isinstance(block, dict) and block.get("type") == "text"
    )

How PiHarness._read_stdout works:

  1. Read lines from proc.stdout (one JSON object per line, same as Claude).
  2. For each line, call _extract_text_pi_live(event) to get live text chunks → append to chunks list (for log streaming).
  3. When "type":"message_end" arrives, call _extract_text_pi_final(event) to get the complete text → set as final_output.
  4. Return HarnessResult(output=final_output or "".join(chunks), ...).

This keeps live logging working (each text_delta is recorded to job.log as it arrives) while also providing the clean, complete message for the control-line extraction (LoopEngine) and handoff.

7.3 Environment Isolation

pi reads configuration from ~/.pi/agent/. _build_env() passes through provider credentials and minimal runtime vars:

def _build_env(self) -> dict:
    safe_keys = {
        "PATH", "HOME", "USER", "SHELL", "LANG",
        "PI_CODING_AGENT_DIR",
    }
    provider_keys = {
        "ANTHROPIC_API_KEY", "OPENAI_API_KEY", "GOOGLE_API_KEY",
        "GOOGLE_GENAI_USE_VERTEXAI", "GOOGLE_GENAI_USE_GENERATIVEAI",
        "GOOGLE_CLOUD_PROJECT", "GOOGLE_CLOUD_LOCATION",
        "AWS_ACCESS_KEY_ID", "AWS_SECRET_ACCESS_KEY",
        "AWS_SESSION_TOKEN", "AWS_REGION",
    }
    safe_keys.update(provider_keys)
    return {k: v for k, v in os.environ.items() if k in safe_keys}

Reproducibility note: Including HOME means pi reads ~/.pi/agent/settings.json, which could contain custom API endpoints, model overrides, or startup preferences. The --no-extensions --no-skills --no-context-files flags disable the main sources of variance, but settings.json can still affect behavior (e.g., quietStartup). This is acceptable for v1 — the same concern exists with claude reading ~/.claude. A future --no-config flag or PI_CONFIG_DIR override could make execution fully hermetic.

7.4 Timeout Handling

Same pattern as ClaudeCodeHarness: threaded stdin/stdout/stderr, wall-clock reader.join(timeout=timeout), and proc.kill() on timeout. pi spawns child processes (bash commands), so proc.kill() alone doesn't terminate the whole tree — the same limitation exists for ClaudeCodeHarness today. Both harnesses should eventually use os.killpg with a new process group (start_new_session=True), but that's a shared improvement tracked separately, not a pi-specific concern.

7.5 Prompt Wrapping

The task prompt is fed via stdin (same approach as ClaudeCodeHarness). pi reads stdin when -p is provided. Same threading pattern:

proc = subprocess.Popen(
    cmd, stdin=subprocess.PIPE, stdout=subprocess.PIPE,
    stderr=subprocess.PIPE, text=True, encoding="utf-8",
    cwd=cwd, env=self._build_env(),
)
threading.Thread(target=self._feed_stdin, args=(proc, prompt), daemon=True).start()

7.6 wrap_prompt (PiHarness)

PiHarness inherits the no-op default. Unlike ClaudeCodeHarness, pi has no harness-specific warnings to inject. The prompt from TaskExecutor is passed through unchanged.
