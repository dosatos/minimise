⚙️ Minimise

A plan orchestrator for multi-agent execution with real-time visualization

Backend-first Multi-job orchestration Context-passing pipeline UI-agnostic

CLI Commands

Job Management

mini job new --plan plan.yaml

Create & start a new job from plan

mini job list

Show all jobs (running/completed/failed)

mini job status <job-id>

Show job details & task progress

Job Control

mini job stop <job-id>

Cancel/stop a running job

mini job resume <job-id>

Retry a failed job from last checkpoint

mini job logs <job-id>

View job output & logs

Visualization & Server

mini view [start]

Launch web UI (defaults to list view, optionally start server)

mini view stop

Shut down the Minimise server

Backend Architecture

Core Components

  • 📄 Plan Parser — reads plan.yaml, validates structure
  • 🎯 Job Manager — creates jobs, tracks multiple concurrent runs
  • ⚡ Task Executor — runs task commands, handles retries (1-3 attempts)
  • 🔗 Handover Manager — passes context between tasks
  • 💾 State Tracker — persists job/task state to disk
  • 🪝 Hook Runner — executes pre/post hooks (plan & task level)
  • 🌐 API Server — REST/WebSocket for UI clients

Execution Loop

┌─────────────────────────────────────────────────────────────┐ │ mini job new --plan plan.yaml │ └──────────────────┬──────────────────────────────────────────┘ ↓ ┌──────────────┐ │ Job Manager │ → creates job_id, init state └──────┬───────┘ ↓ ┌──────────────┐ │ Hook Runner │ → run pre_plan_hook └──────┬───────┘ ↓ ┌────────────────────────────┐ │ Loop: for each task │ │ │ │ ┌────────────────────────┐ │ │ │ Pre-task Hook │ │ │ └────────────────────────┘ │ │ ┌────────────────────────┐ │ │ │ Task Executor │ │ (1-3 retries on error) │ │ - Pass to Claude Code │ │ │ │ - Capture output │ │ │ └────────────────────────┘ │ │ ┌────────────────────────┐ │ │ │ Handover Manager │ │ │ │ - Prepare context │ │ │ │ - For next task │ │ │ └────────────────────────┘ │ │ ┌────────────────────────┐ │ │ │ Post-task Hook │ │ │ └────────────────────────┘ │ │ ┌────────────────────────┐ │ │ │ State Tracker │ │ │ │ - Persist to disk │ │ │ │ - Broadcast via API │ │ │ └────────────────────────┘ │ └────────────────────────────┘ ↓ ┌──────────────┐ │ Hook Runner │ → run post_plan_hook └──────┬───────┘ ↓ ┌──────────────────────┐ │ Job Complete │ │ UIs show final state │ └──────────────────────┘

plan.yaml Structure

Plan Definition

plan:
  name: "Implementation Plan"
  briefing: |
    High-level overview of what this plan does,
    key objectives, and success criteria
  documentation: |
    Reference materials, design docs, links
    that provide context for tasks

  pre_plan_hook: "echo 'Starting plan execution'"
  post_plan_hook: "echo 'Plan execution complete'"

  tasks:
    - id: "task-1"
      name: "Setup Database"
      description: "Initialize database schema and seed data"
      pre_task_hook: "echo 'Setting up DB'"
      post_task_hook: "echo 'DB ready'"

    - id: "task-2"
      name: "Build API"
      description: "Implement REST API endpoints"
      pre_task_hook: ""
      post_task_hook: ""

    # ... more tasks

Key Fields

  • hooks — Shell commands or callbacks at plan/task boundaries (pre_task_hook, post_task_hook)
  • id — Unique task identifier for tracking
  • name, description — Task details for agents
  • 📌 Handover defaults to prompt — Output from previous task passed as context to next agent

Integration with Claude Code

Task Execution Flow

When a task is ready to execute:

1. Minimise constructs payload: - Task definition (name, description, id) - Handover context (output from previous task) - Plan briefing & documentation - Task sequence info 2. Minimise invokes Claude Code: claude-code -p "<payload>" 3. Claude Code executes task: - Receives payload in non-interactive mode - Performs work (implementation, analysis, etc.) - Returns result 4. Minimise captures result: - Extracts task output - Prepares handover context - Stores in State Tracker - Broadcasts via API Server 5. Next task receives handover: - Previous output becomes input context - Process repeats

Handover Strategies

  • prompt — Output sent as a natural prompt/context to next agent
  • command — Output used to construct a shell command for next task
  • structured — Output parsed and passed as structured data (JSON/YAML)

Backend API & Multi-UI Architecture

  • GET /jobs — List all jobs
  • POST /jobs — Create new job
  • GET /jobs/{job-id} — Job details + task list
  • GET /jobs/{job-id}/tasks/{task-id} — Task details, output, retries, diffs

State Persistence & Storage

Local Storage

All state stored locally in ~/.minimise/:

~/.minimise/ ├── minimise.db (SQLite database for all metadata) │ ├── jobs table (job_id, name, status, timestamps) │ ├── tasks table (task_id, job_id, status, output, retries) │ └── diffs table (task_id, diff_data, base_commit) ├── jobs/ │ └── {job-id}/ │ ├── plan.yaml (copy of original plan) │ └── tasks/ │ └── {task-id}/ │ └── diff.txt (git diff output) ├── server.log (API server logs) └── config.json (Minimise configuration)

Concurrency & Atomicity (via SQLite)

Multiple Minimise processes access metadata safely through SQLite:

  • SQLite ACID guarantees — All metadata operations are atomic and consistent
  • Built-in locking — SQLite handles reader/writer coordination automatically
  • No file corruption — SQLite prevents race conditions on metadata writes
  • Job execution safety — Use database transaction to atomically check if job is already running and mark it as running, preventing accidental double-execution

Git-Based Diff Tracking

Pre-Execution Validation

When mini job new --plan plan.yaml starts:

  • ✅ Validate git state: no uncommitted changes, no staged changes
  • ✅ Verify all local work is committed/checked in
  • ✅ If git state is dirty, fail with clear error message
  • ✅ Record base commit hash at job start

Per-Task Diff Calculation

As each task executes:

  • 📌 Keep base commit hash in memory/state.json
  • 📌 After task completes, calculate: git diff base-commit..HEAD
  • 📌 Store diff in ~/.minimise/jobs/{job-id}/tasks/{task-id}/diff.txt
  • 📌 Next task receives: previous task output + diff summary
